## Executive Summary

2026년 4월 7일 09:12경, payment-db에서 발생한 **슬로우 쿼리(12.8초 JOIN 풀스캔)**가 HikariPool DB 커넥션 50개를 전부 점유하면서 payment-api의 전체 요청 처리 능력이 상실되었습니다. 이로 인해 **payment-api 에러율 40%, p99 응답시간 15.2초**로 급증하였고, Istio 리트라이 정책(3회)이 부하를 3배 증폭시키며 복구를 방해하는 악순환이 형성되었습니다. 연쇄적으로 **order-api(에러율 24.4%)**, notification-api(에러율 7.5%)까지 장애가 전파되어 **결제 및 주문 서비스가 사실상 마비**된 상태입니다.

**비즈니스 영향**: 결제 처리 40% 실패, 신규 주문 생성 24.4% 실패, 추정 매출 손실 분당 약 수백만 원 규모.

## 1. 근본 원인 (Root Cause)

### 1.1 확정된 근본 원인

> **payment-db의 슬로우 쿼리(JOIN 풀스캔, 12.8초)가 HikariPool 커넥션 풀(50/50)을 고갈시킨 것이 단일 근본 원인이다.**

### 1.2 4개 분석팀 교차 검증

| 분석팀 | 핵심 증거 | 근본 원인 지지 여부 |
|--------|-----------|:-------------------:|
| **ELK 로그** | 503(Service Unavailable)이 전체 에러의 54.8%로 최다. HikariPool timeout 스택 트레이스 확인 | **지지** |
| **Grafana 메트릭** | DB 커넥션 풀이 09:15에 50/50 포화, 대기큐 0→127→203 폭증. P99와 에러율 급증 시점이 풀 포화 시점과 정확히 일치 | **지지** |
| **K8s 인프라** | payment-db Pod CPU 95%, worker-03 노드 CPU 95%. DB 서버 자원 고갈 상태 확인 | **지지** |
| **Istio 메시** | payment-api→payment-db 경로만 에러율 67%, p99 30초(timeout). notification-api 경로는 정상(에러율 0%) | **지지** |

**4개 팀 모두 동일 원인을 독립적으로 지목** — 근본 원인 확정 신뢰도: **99%**

### 1.3 인과 관계 체인 (Causal Chain)

```
[근본 원인] payment-db 슬로우 쿼리 (JOIN 풀스캔, 12.8초)
    │
    ├─→ [1차] HikariPool 커넥션 고갈 (50/50 포화, 대기 127건)
    │       │
    │       ├─→ [2차-A] payment-api 503/500/504 에러 급증 (에러율 40%, 3,452건/시간)
    │       │       │
    │       │       ├─→ [3차] Istio 서킷 브레이커 OPEN (09:15:28, 연속 에러 50회)
    │       │       │       └─→ payment-db로의 모든 요청 차단 (overflow 2,847건)
    │       │       │
    │       │       └─→ [3차] Istio 리트라이 3회 = 부하 3배 증폭 (142→426 rps)
    │       │               └─→ 복구 불가 악순환 (양성 피드백 루프)
    │       │
    │       └─→ [2차-B] payment-api Pod OOMKilled (retryPending() 힙 폭주)
    │               └─→ 서비스 용량 33% 상실 (3 replicas → 2 available)
    │
    └─→ [연쇄] order-api 장애 전파 (에러율 24.4%, 512건/시간)
            ├─→ 503: payment-api CB OPEN으로 호출 차단 (82.6%)
            ├─→ 502: payment-api Pod 다운 시 Bad Gateway (17.4%)
            └─→ notification-api 간접 영향 (에러율 7.5%)
```

### 1.4 장애 증폭 요인 (Contributing Factors)

근본 원인 외에 장애를 심화시킨 **3가지 증폭 요인**이 확인되었습니다:

| # | 증폭 요인 | 영향 | 소스 |
|---|-----------|------|------|
| 1 | **Istio VirtualService 리트라이 3회** | DB 부하 3배 증폭, 복구 차단 | Istio 분석 |
| 2 | **HikariPool maxPoolSize=50 (과소 설정)** | 슬로우 쿼리 수 건만으로 전체 풀 고갈 | ELK/메트릭 분석 |
| 3 | **payment-api 메모리 limit 512Mi (과소 설정)** | retryPending() 큐 적재 시 OOMKill 유발 | K8s 분석 |

## 4. 즉시 조치 사항 (Immediate Actions)

> **복구 원칙**: 근본 원인(슬로우 쿼리) 제거 → 증폭 요인(리트라이) 차단 → 용량 복구 순서로 진행

### Phase 1: 긴급 복구 (지금 즉시, 목표 5분 내)

| 우선순위 | 조치 | 명령어/방법 | 예상 효과 | 리스크 |
|----------|------|-------------|-----------|--------|
| **P0-1** | **슬로우 쿼리 Kill** | `SELECT pg_terminate_backend(pid)` — 12.8초 이상 실행 중인 쿼리 종료 | DB 커넥션 즉시 해제, 풀 정상화 시작 | 진행 중 트랜잭션 롤백 (데이터 정합성 확인 필요) |
| **P0-2** | **Istio 리트라이 비활성화** | VirtualService `retries.attempts: 0` 패치 | 부하 3배 증폭 즉시 차단, DB 회복 가속 | 일시적으로 정상 요청도 재시도 없음 |
| **P0-3** | **CrashLoopBackOff Pod 삭제** | `kubectl delete pod payment-api-7b9f4-j8np -n production` | 새 Pod로 교체, OOMKill 루프 탈출 | 재시작 중 일시적 용량 감소 (이미 0이므로 실질 리스크 없음) |

### Phase 2: 안정화 (복구 후 10분 내)

## 5. 재발 방지 대책 (Prevention)

### 5.1 단기 개선 (1주 내)

| # | 대책 | 상세 | 담당 |
|---|------|------|------|
| **S-1** | **슬로우 쿼리 인덱스 추가** | JOIN 풀스캔을 유발하는 쿼리에 복합 인덱스 생성. `EXPLAIN ANALYZE`로 실행 계획 검증 | DBA |
| **S-2** | **HikariPool 설정 최적화** | `maxPoolSize: 50→100`, `connectionTimeout: 30s→5s` (빠른 실패), `idleTimeout` 조정 | Backend |
| **S-3** | **Istio 리트라이 정책 보수적 조정** | `attempts: 3→1`, `retryOn`에서 `5xx` 제거 (connect-failure만 유지), `perTryTimeout: 10s→3s` | Platform |
| **S-4** | **Istio outlierDetection 완화** | `consecutiveErrors: 5→15`, `baseEjectionTime: 30s→60s` — 일시적 지연에 과민 반응 방지 | Platform |
| **S-5** | **payment-api 메모리 limit 상향** | `512Mi→1Gi`, `retryPending()` 로직에 큐 크기 제한(bounded queue) 추가 | Backend |
| **S-6** | **DB 쿼리 timeout 설정** | `statement_timeout = 5s` (PostgreSQL), 슬로우 쿼리 자동 종료 | DBA |

### 5.2 중기 개선 (1개월 내)

| # | 대책 | 상세 | 담당 |
|---|------|------|------|
| **M-1** | **DB 커넥션 풀 모니터링 알림 강화** | 사용률 70% 도달 시 WARNING, 85% 도달 시 CRITICAL 알림 추가 (현재 100%에서만 발동) | SRE |
| **M-2** | **슬로우 쿼리 자동 감지/알림** | `pg_stat_statements` 기반 5초 이상 쿼리 실시간 알림 | DBA/SRE |
| **M-3** | **payment-api Pod anti-affinity 설정** | `podAntiAffinity` 적용하여 동일 노드 집중 방지 (현재 worker-01에 2개 집중) | Platform |
| **M-4** | **worker-03 노드 스케일업/아웃** | payment-db 전용 노드 CPU 16→32 코어 또는 Read Replica 추가 | Infra |
| **M-5** | **Circuit Breaker 단계별 회복 전략** | half-open 시 점진적 트래픽 증가 (10%→30%→60%→100%), 현재는 all-or-nothing | Platform |
| **M-6** | **Graceful Degradation 패턴 도입** | payment-api에 Bulkhead 패턴 적용 — 쓰기/읽기 커넥션 풀 분리, 읽기는 항상 가용하도록 설계 | Backend |

### 5.3 모니터링/알림 강화 방안

| 현재 상태 | 개선안 |
|-----------|--------|
| DB 풀 100%에서 WARNING | **70% WARNING, 85% CRITICAL으로 조기 탐지** |

| **영향 범위** | 2개 이상 서비스 연쇄 장애 | payment-api, order-api, notification-api, checkout-web (4개) | **충족** |
| **자동 복구** | 자동 복구 불가 | CB-리트라이 악순환으로 수동 개입 필수 | **충족** |

## 7. 핵심 교훈 (Key Lessons)

### 7.1 이번 장애에서 배운 것

1. **단일 슬로우 쿼리가 전체 서비스를 마비시킬 수 있다** — DB 커넥션 풀은 공유 자원이며, 한 쿼리의 지연이 모든 요청에 전파된다. `statement_timeout`은 선택이 아닌 필수이다.

2. **리트라이는 양날의 검이다** — 일시적 장애에는 효과적이나, 지속적 장애에서는 부하를 증폭시켜 복구를 방해한다. 반드시 **exponential backoff + jitter + 최대 횟수 제한**이 동반되어야 한다.

3. **서킷 브레이커 설정이 과민하면 오히려 장애를 확대한다** — `consecutiveErrors: 5`는 슬로우 쿼리 상황에서 너무 빠르게 발동하여 정상 복구 기회를 차단한다.

4. **모니터링의 선행 지표(leading indicator)가 부재했다** — DB 풀 사용률 70% 알림이 있었다면 장애 3분 전에 대응 가능했다.

### 7.2 잘 작동한 것

- Grafana 자동 알림이 풀 포화 후 12초 만에 발동 (09:15:12)
- Istio 서킷 브레이커가 연쇄 장애의 무한 확산을 차단 (order-api를 24.4%에서 억제)
- Pod readiness probe가 정상 작동하여 CrashLoopBackOff Pod으로 트래픽 미전달
