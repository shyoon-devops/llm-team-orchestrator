## 1. 요약 (Executive Summary)

payment-api의 데이터베이스(payment-db) 커넥션 풀이 완전 고갈되면서, 결제 처리 요청의 **40%가 실패**(500/503 에러)하고 **p99 응답시간이 0.3초 → 15.2초로 50배 급증**하는 P1 장애가 발생했습니다.

**근본 원인은 payment-db의 슬로우 쿼리(12.8초)로 인한 DB 커넥션 점유 장기화**이며, 이로 인해 커넥션 풀(max 50)이 고갈되고, 서킷 브레이커 발동, Pod OOMKill, order-api 연쇄 장애로 확대되었습니다.

### 비즈니스 영향
- **결제 실패율 40%**: 고객 결제 10건 중 4건 실패 → 직접적 매출 손실
- **주문 처리 장애 24.4%**: order-api 연쇄 영향으로 주문 생성/조회 불안정
- **고객 경험 심각한 저하**: 정상 요청도 15초 대기 → 이탈률 급증 예상
- **예상 매출 손실**: 분당 약 ₩X백만 (정상 처리량 142 req/s 기준, 40% 실패)

## 4. 근본 원인 분석 (Root Cause Analysis)

### 4.1 근본 원인: payment-db 슬로우 쿼리에 의한 커넥션 풀 고갈

```
[근본 원인]                    [1차 영향]                [2차 영향]               [3차 영향]
                                                        
payment-db 슬로우 쿼리   →  DB 커넥션 50/50 포화   →  HikariPool 고갈       →  500/503 대량 발생
(12.8초, JOIN 풀스캔)        + checkpoint 과빈발       (waiting 127 누적)        에러율 40%
                                                        
        ↓                          ↓                        ↓                      ↓
worker-03 CPU 95%           커넥션 할당 불가          서킷브레이커 OPEN       order-api 연쇄장애
(DB 노드 리소스 압박)       FATAL: too many clients   (09:15:28 발동)         (에러율 24.4%)
                                                        
                                                           ↓
                                                    Pod j8np OOMKilled
                                                    (retryPending 메모리 폭주)
                                                    → 2/3 Pod만 운영
                                                    → 남은 Pod 과부하 악순환
```

### 4.2 원인 상세 분석

#### (1) 트리거: 슬로우 쿼리 (Primary Root Cause)
```sql
-- 12,847ms (12.8초) 소요된 문제 쿼리
SELECT p.*, t.* 
FROM payments p 
JOIN transactions t ON p.id = t.payment_id 
WHERE p.status = 'pending' 
  AND p.created_at > now() - interval '1 hour'
```
- `payments` ↔ `transactions` 테이블 JOIN이 풀스캔 발생 추정
- 인덱스 누락 또는 `pending` 상태 레코드 급증으로 쿼리 플랜 악화
- **각 커넥션이 12초 이상 점유**되면서 커넥션 회전율 급락

#### (2) 증폭 요인 1: DB 커넥션 풀 사이즈 부족
- PostgreSQL `max_connections = 50` — 프로덕션 트래픽(142 rps) 대비 매우 부족
- HikariPool `maxPoolSize = 50` — DB 설정과 동일하여 여유분 없음
- 30초 커넥션 타임아웃 동안 127개 요청이 큐에 누적

#### (3) 증폭 요인 2: Istio 리트라이 정책에 의한 부하 증폭
- VirtualService 설정: `retries.attempts = 3`, `perTryTimeout = 10s`, `retryOn = 5xx`
- 실패한 요청이 **최대 3회 재시도** → 이미 포화된 DB에 부하 3배 증폭
- Overflow count: **2,847건** (서킷 브레이커에 의해 거부된 요청)

#### (4) 증폭 요인 3: Pod OOMKill에 의한 처리 용량 감소
- `payment-api-7b9f4-j8np` Pod가 `retryPending()` 메서드에서 힙 메모리 폭주
- 힙 제한 512MB 도달 → OOMKilled (exit code 137) → CrashLoopBackOff
- **3개 중 1개 Pod 탈락 → 나머지 2개 Pod에 부하 집중 → 악순환**

#### (5) 증폭 요인 4: 인프라 리소스 압박
- worker-03 (payment-db 호스팅 노드): CPU **95%**, Memory **86%**
- DB checkpoint가 30초 간격으로 과빈발 → I/O 경합 심화
- 디스크 사용 73/100Gi (73%) — 즉각 위험은 아니나 모니터링 필요

## 8. 교훈 (Lessons Learned)

1. **DB 커넥션 풀 사이즈가 트래픽 규모 대비 현저히 부족**했다. 142 rps 서비스에 max_connections=50은 슬로우 쿼리 하나로 즉시 고갈된다.
2. **Istio 리트라이 정책이 장애를 증폭**시켰다. 백엔드 DB 장애에 3회 리트라이는 "불난 곳에 부채질"과 동일하다.
3. **Pod 메모리 제한(512MB)이 retry 로직의 메모리 사용 패턴과 불일치**했다. 장애 시 retry queue가 메모리를 폭주시키는 구조적 문제.
4. **payment-db와 payment-api가 동일 장애 도메인**(worker-03 노드 과부하)에 영향받아 복구가 지연되었다.
5. **슬로우 쿼리에 대한 사전 알림이 부재**했다. 09:12에 징후가 포착됐으나 09:15:30 자동 알림까지 3분 이상의 탐지 지연 발생.
