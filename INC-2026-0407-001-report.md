## 1. Executive Summary (경영진 요약)

payment-api 서비스의 데이터베이스 커넥션 풀이 완전히 고갈되면서 **결제 처리 서비스에 전면 장애**가 발생했습니다.
에러율이 평소 0.1%에서 **최대 67.1%**까지 치솟았고, 응답 시간은 정상(0.3초)의 **54배인 15.2초**에 달했습니다.
이 장애는 주문(order-api), 체크아웃(checkout-web), 알림(notification-api) 서비스로 **연쇄 전파**되어
사실상 **결제가 수반되는 모든 비즈니스 흐름이 40~67% 실패** 상태에 놓였습니다.

### 비즈니스 영향

- **결제 실패율**: 약 40~67% (시간대에 따라 변동)
- **예상 매출 손실**: 장애 지속 시간 × 평균 결제 처리량 기준 산정 필요
- **고객 영향**: 결제 시도 고객의 약 절반이 실패 또는 15초 이상 대기 경험
- **주문 처리 실패**: order-api 에러율 24.4%, 결제 연동 주문 흐름 전면 영향

## 3. Root Cause Analysis (근본 원인 분석)

### 3.1 근본 원인 (Root Cause)

**payment-db(PostgreSQL) 인스턴스의 리소스 포화가 장애의 진원지이다.**

payment-db가 위치한 worker-03 노드의 CPU 사용률이 **95%**, 메모리 사용률이 **87%**에 달하면서
DB 쿼리 응답 시간이 극도로 느려졌다. 이로 인해 payment-api의 JDBC 커넥션이 반환되지 않아
HikariPool 커넥션 풀(max=50)이 완전히 고갈되었으며, 이것이 전체 장애의 기폭제가 되었다.

### 3.2 장애 인과 관계 체인 (Causal Chain)

```
[1단계: 기폭제]
  payment-db 리소스 포화 (CPU 95%, Memory 87%)
  → DB 쿼리 응답 지연 / slow query 발생
      │
[2단계: 커넥션 풀 고갈]
  │→ JDBC 커넥션 미반환 → HikariPool 50/50 포화 → 대기큐 127→203건
  │→ 커넥션 획득 실패 (30초 타임아웃) → HTTP 500 에러 대량 발생
      │
[3단계: 장애 증폭 — 3가지 증폭 요인이 동시 작용]
  │
  ├─ (A) Istio retry 정책 증폭
  │   VirtualService retry 3회 × perTryTimeout 10초
  │   → 실패 요청마다 3배 트래픽 증폭 → 과부하 가속
  │
  ├─ (B) Pod OOMKill로 용량 감소
  │   payment-api-j8np: PaymentProcessor.retryPending()에서
  │   재시도 큐 메모리 폭증 → Java heap 512MB 초과 → OOMKilled
  │   → 3대 중 1대 탈락 → 남은 2대에 트래픽 집중 (CPU 78~82%)
  │
  └─ (C) 서킷 브레이커 flapping
      ejection 시간 30초로 짧아 OPEN↔HALF_OPEN 반복
      → 안정적 fail-fast 불가 → 불필요한 DB 연결 재시도 지속
      │
[4단계: Cascade Failure (연쇄 전파)]
  │→ payment-api 에러율 67.1%, p99 15.2초
  │→ order-api → payment-api 호출 실패 (에러율 43%, 타임아웃 15초)
  │→ checkout-web → payment-api 호출 실패 (에러율 35%)
  │→ notification-api 영향 시작 (에러율 7.5%)
```

## 6. Lessons Learned (교훈)

### 6.1 잘된 점 (What Went Well)

- 서킷 브레이커가 발동하여 payment-db에 대한 무제한 연결 시도를 차단, 완전한 시스템 다운을 방지
- Grafana/Prometheus 알림 체계가 장애 발생 후 18초 이내에 최초 알림(DBConnectionPoolExhausted)을 발동
- 4개 분석팀의 병렬 조사로 10분 이내에 근본 원인을 교차 검증으로 확정

### 6.2 개선이 필요한 점 (What Needs Improvement)

1. **조기 감지 실패**: 09:10~09:15 사이 5분의 열화 구간에서 커넥션 풀이 24%→100%로 증가했으나, WARNING 알림은 100% 도달 시에만 발동. 60~80% 단계의 조기 경보가 없었음
2. **Retry 정책의 역효과**: 정상 상황을 전제로 한 retry 3회 설정이 장애 시 트래픽을 3배로 증폭하여 복구를 방해
3. **단일 장애점(SPOF)**: payment-db가 단일 호스트로 운영되어, 1대의 장애가 전체 결제 파이프라인을 마비시킴
4. **메모리 설정 부적절**: payment-api의 Java heap 512MB가 재시도 큐 증가 시나리오에 불충분
5. **장애 전파 차단 미비**: order-api에 서킷 브레이커/fallback이 없어 payment-api 장애가 그대로 전파

### 6.3 핵심 교훈

> **"정상 시에 유용한 설정(retry, 긴 timeout)이 장애 시에는 최악의 증폭기가 된다."**
> 모든 resilience 패턴은 장애 시나리오에서의 동작을 반드시 함께 검증해야 한다.
